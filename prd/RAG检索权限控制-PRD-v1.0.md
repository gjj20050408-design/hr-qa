## 2. 需求背景与动机

### 2.1 问题陈述

当前 PRD V1.0 中，检索权限仅区分两级：

| 文档状态 | 可见范围 | 问题 |
|----------|----------|------|
| `已发布` | 全员可检索 | **粒度太粗**：薪酬制度、绩效制度等敏感内容同样全员可见 |
| `已归档` | 仅HR可见 | 归档意味着"制度已废止"，而非"制度敏感" |

**核心矛盾**：一份正在生效的《薪酬结构细则》必须是 `已发布` 状态，但其内容（薪资等级、奖金计算规则）不应向全员开放检索。

### 2.2 典型痛点场景

| 场景 | 用户 | 问题描述 |
|------|------|----------|
| 🔴 敏感薪酬查询 | 普通员工 | 在RAG中问"P7级别的薪资范围是多少？"，系统检索到《薪酬结构细则》并返回具体数字 |
| 🔴 晋升条件泄露 | 普通员工 | 搜索"晋升条件"，直接检索到《管理晋升评估细则》中的内部评审标准 |
| 🟡 绩效公式曝光 | 普通员工 | 问"绩效奖金怎么算的"，RAG返回了仅HR应知的详细计算公式 |

### 2.3 期望效果

- ✅ 敏感制度保持 `已发布` 状态（因为正在生效），但检索权限可按角色控制
- ✅ 普通员工检索时，自动过滤无权限文档，保障敏感信息不泄露
- ✅ HR 可在后台灵活决定哪些分类/文档需要限制，无需开发介入
- ✅ 权限过滤逻辑对关键词搜索和RAG问答**统一生效**

## 3. 核心设计决策

本节记录了经与用户确认的4项核心决策，作为需求设计的基石。

| # | 决策项 | 结论 | 理由 |
|---|--------|------|------|
| **D1** | 权限控制粒度 | **两者结合**：文档默认继承所属分类的权限，但单份文档可单独覆盖 | 兼顾灵活性与管理便利性——分类设默认规则，特殊文档单独处理 |
| **D2** | 敏感文档范围 | **由HR在后台自主标记**，不做预设 | 不同企业的敏感边界不同，硬编码会造成不适配 |
| **D3** | 无权限时的处理策略 | **明确拒绝**：告知用户"该内容需要HR权限才能查看" | 用户明确知晓信息存在但不可访问，避免"系统坏了"的误解；需联系HR时目标清晰 |
| **D4** | 权限过滤范围 | **统一过滤**：关键词搜索和RAG问答均受权限控制 | 安全策略必须一致，不能留"后门"——搜索搜不到、RAG也问不出 |

## 4. 功能性需求

### 4.1 新增功能：检索权限管理（HR端）

#### FR-2.6 分类检索权限设置

- **描述**：HR专员可为每个文档分类设置默认检索权限，该分类下新建文档自动继承
- **权限级别定义**：

| 级别 | 标识 | 可检索角色 | 典型场景 |
|------|------|------------|----------|
| **全员可见** | `all_roles` | employee、hr_specialist、admin | 考勤制度、休假制度等通用制度 |
| **仅HR+管理员** | `hr_admin_only` | hr_specialist、admin | 薪酬结构、绩效评估细则、晋升标准 |
| **仅管理员** | `admin_only` | admin | 系统配置相关文档、审计策略 |

- **操作入口**：HR管理后台 → 文档管理 → 分类管理 → 右键/操作菜单 →「设置检索权限」
- **交互规则**：
  - 新建分类默认 `all_roles`（全员可见）
  - 修改分类权限后，弹出确认对话框：
    > 「是否将新权限应用到该分类下所有**继承权限**的文档？」
    > [是，全部应用]  [否，仅影响新建文档]
  - 选择"是"：批量更新所有 `access_level = inherit` 的子文档的实际可见范围
  - 选择"否"：仅修改分类默认值，已有文档保持原状
- **约束**：
  - 仅 HR专员和管理员可操作
  - `admin_only` 级别仅管理员可设置

#### FR-2.7 文档检索权限覆盖

- **描述**：HR专员可为单份文档单独设置检索权限，覆盖其所属分类的默认权限
- **文档权限级别**：

| 级别 | 标识 | 说明 |
|------|------|------|
| **继承分类** | `inherit` | 默认值，跟随所属分类的权限设置 |
| **全员可见** | `all_roles` | 覆盖分类限制，强制全员可见 |
| **仅HR+管理员** | `hr_admin_only` | 覆盖分类默认，限制为HR及以上 |
| **仅管理员** | `admin_only` | 覆盖分类默认，限制为仅管理员 |

- **操作入口**：
  - 方式一：文档列表 → 操作菜单 →「权限设置」→ 选择权限级别
  - 方式二：文档上传/编辑页面 →「检索权限」下拉框（默认选中"继承分类"）
- **交互提示**：
  - 当文档权限与分类权限不一致时，在文档列表的权限列显示**橙色标记**（如 `⚠ 已覆盖`），鼠标悬停显示：
    > 分类默认：全员可见 → 本文档：仅HR+管理员
- **约束**：
  - `admin_only` 级别仅管理员可设置
  - 不允许将权限放宽到超过操作者自身角色（HR不能将文档设为 `admin_only`）

#### FR-1.6 检索权限拦截提示（所有角色）

- **描述**：当用户检索（搜索/RAG）时，系统根据其角色自动过滤无权限文档，并在特定场景下给出明确提示
- **处理策略（3种场景）**：

| 场景 | 条件 | 系统行为 |
|------|------|----------|
| **全部命中** | 检索结果全部来自有权限的文档 | 正常返回结果，用户无感知 |
| **部分过滤** | 部分结果被权限过滤，仍有可展示的结果 | 正常返回有权限的结果；结果区底部显示灰色提示条：`🔒 部分相关制度需要HR权限才能查看，如需了解请联系HR部门` |
| **全部被过滤** | 所有匹配结果均来自无权限文档 | **不返回空结果**，而是显示明确提示页面：`🔒 该内容需要HR权限才能查看。如有疑问，请联系HR部门（hr@company.com）` |

- **RAG问答特殊处理**：
  - 场景"全部被过滤"：不调用LLM（避免浪费Token），直接返回权限提示
  - 场景"部分过滤"：仅使用有权限的文档片段构建Prompt；回答末尾附加灰色提示文字
- **日志记录**：
  - 每次权限过滤事件写入审计日志（`audit_logs`），记录被过滤的文档ID列表
  - 用于后续分析：哪些制度被频繁"挡住"，可能需要HR评估是否应放宽权限

### 4.2 已有功能修改

#### FR-3.1 关键词全文搜索（修订）

在原 FR-3.1 基础上**增加权限过滤步骤**。修改后的处理逻辑：

```
① 对输入进行分词处理
    │
    ▼
② 在文档索引中匹配标题和正文
    │
    ▼
③ 【新增】权限过滤：根据当前用户角色，排除无权限文档 ← 新增步骤
    │
    ▼
④ 按相关度排序返回结果
```

- 搜索结果展示不变（标题 + 摘要 + 高亮 + 分类标签）
- 过滤后的结果数量在搜索栏下方显示，如 `找到 12 条结果（已过滤 3 条受限文档）`
- 性能要求不变：搜索响应时间 < 1秒

#### FR-4.2 精准回答生成（修订）

在原 FR-4.2 的 RAG 处理流程中**增加权限过滤步骤**。修改后的处理流程：

```
用户提问
    │
    ▼
① 问题向量化（Embedding）
    │
    ▼
② 向量检索 → Top-K 相关制度片段（K=10，比实际需要的略多，为过滤留余量）
    │
    ▼
③ 【新增】权限过滤：逐一检查文档访问权限，排除无权限文档 ← 新增步骤
    │
    ├── 全部被过滤 → 返回权限拒绝提示，不调用LLM
    │
    ▼（有可用文档）
④ 取过滤后的 Top-N（N≤5）构建 Prompt
    │
    ▼
⑤ LLM 生成回答（必须引用出处）
    │
    ▼
⑥ 返回：回答正文 + 参考条文链接 + 【部分过滤时附加提示】
```

- **向量检索策略调整**：
  - 原方案：检索 Top-5 片段
  - 新方案：检索 Top-10 片段 → 权限过滤 → 取 Top-5（留余量避免过滤后不够）
- **权限过滤的片段级处理**：
  - 检索返回的每个Chunk都关联其所属文档ID
  - 根据文档ID查询权限，而非根据Chunk内容判断
  - 同一文档的不同Chunk共享同一权限判定

| §4.4 高级智能问答（RAG） | 🔴 高 | 修改 FR-4.2 | RAG检索流程增加权限过滤；调整Top-K策略 |

| 其余章节 | 🟢 无 | — | 不受影响 |

### 5.2 一期→二期过渡设计的影响

原PRD中预留的抽象层设计**完全兼容**本次变更：

| 预留项 | 兼容性 | 说明 |
|--------|:------:|------|
| `DocumentChunkService.split()` | ✅ | 权限过滤发生在Chunk检索之后，与Chunk切分逻辑解耦 |
| `EmbeddingProvider` | ✅ | 不受影响 |
| `VectorStore` | ✅ | 需在 `query()` 方法中增加 `role` 参数，或在上层做后置过滤 |
| 问答策略链 `RAGHandler` | ✅ | 权限过滤可作为独立 `PermissionFilter` 插入策略链 |
| 文档预留字段 `chunk_count`、`embedding_status` | ✅ | 新增的 `access_level` 字段与预留字段不冲突 |

### 5.3 策略链演进

```
一期（原）:
  FAQ匹配 → 规则匹配 → 全文搜索 → [RAG槽位]

一期（修订后）:
  FAQ匹配 → 规则匹配 → [权限过滤] → 全文搜索 → [RAG槽位]

二期（原）:
  FAQ匹配 → 规则匹配 → 全文搜索 → RAG智能问答

二期（修订后）:
  FAQ匹配 → 规则匹配 → [权限过滤] → 全文搜索 → [权限过滤] → RAG智能问答
                        ↑                              ↑
                   搜索阶段过滤                   RAG向量检索后过滤
```

> **设计原则**：权限过滤作为横切关注点（Cross-Cutting Concern），在策略链中集中处理，不分散到各个策略内部。

## 8. 数据需求变更

### 8.1 分类标签表（categories）— 新增字段

| 字段名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|:----:|--------|------|
| `access_level` | ENUM('all_roles','hr_admin_only','admin_only') | ✅ | `all_roles` | 该分类下文档的默认检索权限级别 |

### 8.2 制度文档表（documents）— 新增字段

| 字段名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|:----:|--------|------|
| `access_level` | ENUM('inherit','all_roles','hr_admin_only','admin_only') | ✅ | `inherit` | 文档检索权限级别；`inherit` 表示继承所属分类的权限 |

### 8.3 权限判定逻辑（伪代码）

```
function canAccess(document, userRole):
    if userRole == 'admin':
        return true                    // 管理员可查看所有文档
    
    effectiveLevel = document.access_level
    if effectiveLevel == 'inherit':
        effectiveLevel = document.category.access_level
    
    switch effectiveLevel:
        case 'all_roles':
            return true                // 全员可见
        case 'hr_admin_only':
            return userRole in ['hr_specialist', 'admin']
        case 'admin_only':
            return userRole == 'admin'
    
    return false
```

### 8.4 权限级别汇总表

| 文档 access_level | 分类 access_level | 最终 effectiveLevel | employee | hr_specialist | admin |
|:---:|:---:|:---:|:---:|:---:|:---:|
| `inherit` | `all_roles` | `all_roles` | ✅ | ✅ | ✅ |
| `inherit` | `hr_admin_only` | `hr_admin_only` | ❌ | ✅ | ✅ |
| `inherit` | `admin_only` | `admin_only` | ❌ | ❌ | ✅ |
| `all_roles` | （任意） | `all_roles` | ✅ | ✅ | ✅ |
| `hr_admin_only` | （任意） | `hr_admin_only` | ❌ | ✅ | ✅ |
| `admin_only` | （任意） | `admin_only` | ❌ | ❌ | ✅ |

> **设计要点**：文档级设置一旦非 `inherit`，则完全覆盖分类默认值，`admin` 角色始终拥有最高权限。

## 9. 验收标准

### 9.1 功能验收

- [ ] **AC-PERM-01**：HR专员可在分类管理页面设置检索权限（全员可见 / 仅HR+管理员 / 仅管理员），保存后生效
- [ ] **AC-PERM-02**：修改分类权限时弹出确认对话框，选择"是"后，该分类下所有 `inherit` 权限的文档同步更新
- [ ] **AC-PERM-03**：HR专员可在单份文档的编辑页覆盖检索权限，覆盖后在文档列表显示"已覆盖"标记
- [ ] **AC-PERM-04**：普通员工关键词搜索时，无权限的文档不出现在搜索结果中
- [ ] **AC-PERM-05**：普通员工RAG提问时，回答仅基于有权限的文档生成，不会引用无权限文档内容
- [ ] **AC-PERM-06**：普通员工搜索/RAG提问时，若所有匹配结果均被过滤，显示明确提示：`该内容需要HR权限才能查看`
- [ ] **AC-PERM-07**：普通员工搜索/RAG提问时，若部分结果被过滤，结果区域底部显示提示：`部分相关制度需要HR权限才能查看`

### 9.2 回归验收（确保已有功能不受影响）

- [ ] **AC-REG-01**：HR专员和管理员搜索/RAG提问时，所有 `已发布` 文档均可检索（不受权限限制）
- [ ] **AC-REG-02**：文档 `已归档` 状态的行为不变：仍然全员不可检索（包括HR通过搜索—仅管理后台可查）
- [ ] **AC-REG-03**：问答记录中不显示无权限文档的引用信息
- [ ] **AC-REG-04**：全文搜索响应时间仍满足 P95 < 1秒（权限过滤不显著增加延迟）

### 9.3 非功能验收

- [ ] **AC-NFR-01**：权限判定在检索阶段完成，不在前端暴露无权限文档的任何信息（包括标题）
- [ ] **AC-NFR-02**：权限过滤事件写入审计日志（记录用户、时间、过滤文档数）
- [ ] **AC-NFR-03**：新建分类/文档的默认权限为"全员可见"（安全默认值，符合"由HR主动标记敏感"的设计）
