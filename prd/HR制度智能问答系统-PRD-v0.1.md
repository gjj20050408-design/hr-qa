## 1. 引言与背景

### 1.1 业务背景

随着公司规模扩大和HR制度体系日益复杂，员工对以下政策的咨询量激增：

- **考勤制度**：打卡规则、加班认定、调休政策
- **薪酬制度**：薪资结构、发放日期、个税计算
- **福利制度**：体检资格、补贴标准、弹性福利
- **休假制度**：年假、病假、婚假、产假天数及条件
- **绩效制度**：考核周期、评分标准、晋升条件

当前主要依赖HR人工解答，存在**五大痛点**：

| 痛点 | 表现 | 影响 |
|------|------|------|
| 信息分散 | 制度散落在邮件、共享盘、多个PDF版本中 | 查找低效，版本混乱 |
| 响应延迟 | HR专员需反复查阅文件，平均回复耗时半天 | 高峰时段工单积压严重 |
| 一致性差 | 不同HR对同一政策解释存在差异 | 员工信任度下降，合规风险 |
| 查询门槛高 | 长篇制度难以快速提取个人相关信息 | 员工体验差 |
| 知识沉淀弱 | 常见问答记录无序 | 无法分析员工真正关心的问题 |

### 1.2 建设目标

构建一套**HR制度智能问答系统**，实现：

- ✅ 员工自助查询，秒级获取准确答案
- ✅ 基于RAG技术，从制度原文生成有出处的回答
- ✅ 个性化权益报告，主动推送员工可享受的福利
- ✅ 数据驱动决策，分析咨询热点优化制度宣导
- ✅ 减少HR重复咨询工作量50%以上

### 1.3 适用范围

本PRD定义系统的功能需求、非功能需求、数据需求、接口需求和验收标准，作为设计、开发和测试的依据文档。

### 1.4 术语与缩写

| 术语 | 全称/说明 |
|------|----------|
| **RAG** | Retrieval-Augmented Generation，检索增强生成 |
| **LLM** | Large Language Model，大语言模型 |
| **FAQ** | Frequently Asked Questions，常见问题解答 |
| **SSO** | Single Sign-On，单点登录 |
| **LDAP** | Lightweight Directory Access Protocol，轻量目录访问协议 |
| **Embedding** | 向量嵌入，将文本转为数学向量表示 |

## 4. 功能性需求

### 4.1 用户与权限管理模块

#### FR-1.1 员工账号注册
- **描述**：支持新员工通过工号/手机号/邮箱注册系统账号
- **输入**：工号（必填）、姓名（必填）、手机号（选填）、邮箱（选填）、密码（必填）
- **验证规则**：
  - 工号：字母数字组合，4-20位，唯一性校验
  - 密码：不少于8位，含大小写字母+数字
  - 邮箱：格式合法性校验
- **一期策略**：独立账号密码体系（课程设计场景，不依赖外部统一认证）
- **二期规划**：对接公司统一认证（SSO/LDAP）— 预留接口，实际对接视企业环境而定

#### FR-1.2 用户登录
- **描述**：已注册用户通过凭证登录系统
- **输入**：账号（工号/手机/邮箱）+ 密码
- **安全机制**：
  - 连续5次失败锁定30分钟
  - 登录态Token有效期2小时，支持Refresh Token续期
  - 敏感操作二次验证（修改密码等）

#### FR-1.3 组织架构绑定
- **描述**：账号自动关联员工所属部门、职级等信息
- **数据来源**：一期通过管理员批量导入Excel组织架构表；二期对接HR系统API
- **绑定规则**：一个员工属于唯一主部门（可关联多个虚拟项目组）

#### FR-1.4 权限分级控制
- **描述**：基于RBAC模型实现细粒度权限控制
- **角色与权限**：详见[3.2 权限矩阵](#32-权限矩阵rbac)
- **实现要求**：
  - 前端：按角色动态渲染菜单和操作按钮
  - 后端：API接口级权限拦截
  - 数据级：员工仅能查看自己的问答记录，HR可查看本部门统计数据

#### FR-1.5 密码管理
- 支持修改密码（需验证旧密码）
- 支持忘记密码→通过绑定邮箱/手机重置

### 4.2 制度文档管理模块

#### FR-2.1 文档上传
- **描述**：HR专员上传HR制度文档到知识库
- **支持格式**：PDF、Word（.doc/.docx）、Markdown、富文本（HTML）
- **上传流程**：
  1. HR选择本地文件
  2. 填写文档标题（必填）、选择分类标签（必填）、填写版本说明（选填）
  3. 系统自动解析文档内容，提取文本
  4. 文档进入知识库，状态为"已发布"

#### FR-2.2 文档分类与标签
- **描述**：对制度文档按分类体系管理
- **预置分类（树形结构）**：

```
制度分类
├── 考勤制度
│   ├── 打卡管理
│   ├── 加班政策
│   └── 调休规定
├── 薪酬制度
│   ├── 薪资结构
│   ├── 奖金发放
│   └── 个税说明
├── 福利制度
│   ├── 健康体检
│   ├── 补贴标准
│   └── 弹性福利
├── 休假制度
│   ├── 年假规定
│   ├── 病假规定
│   ├── 婚假/产假
│   └── 其他假期
└── 绩效制度
    ├── 考核周期
    └── 晋升条件
```
- HR可自定义新增/修改分类节点

#### FR-2.3 版本控制
- **描述**：自动保留文档历史版本，支持版本回溯
- **版本号规则**：主版本.次版本（如 V1.0 → V2.0 为主更新，V1.0 → V1.1 为修正）
- **功能点**：
  - 每次更新自动生成新版本，旧版本归档保留
  - 支持任意两个版本差异对比（高亮变更内容）
  - 支持回滚到任意历史版本
  - 更新时展示变更摘要，触发通知（二期联动通知模块）

#### FR-2.4 文档检索
- **描述**：HR可多维度检索知识库中的制度文档
- **检索维度**：
  - 按标题模糊搜索
  - 按分类/标签筛选
  - 按发布时间范围筛选
  - 按版本号筛选
  - 按文档状态筛选（已发布/草稿/已归档）

#### FR-2.5 文档状态管理

| 状态 | 说明 | 操作 |
|------|------|------|
| 草稿 | 仅上传者可见，尚未发布 | 编辑、发布、删除 |
| 已发布 | 全员可检索和查看 | 更新（生成新版本）、归档 |
| 已归档 | 不再对员工可见，仅HR和历史记录中可查 | 恢复 |

### 4.3 智能检索与基础问答模块

#### FR-3.1 关键词全文搜索
- **描述**：员工输入关键词，在全部已发布制度文档中全文检索
- **输入**：搜索关键词（1-100字符）
- **处理逻辑**：
  1. 对输入进行分词处理
  2. 在文档索引中匹配标题和正文
  3. 按相关度排序返回结果
- **搜索结果展示**：
  - 文档标题 + 命中摘要片段
  - 命中关键词**高亮显示**
  - 点击可定位至原文段落（锚点跳转）
  - 显示文档分类标签和版本信息
- **性能要求**：搜索响应时间 < 1秒

#### FR-3.2 FAQ管理与匹配
- **描述**：HR预设高频问题的标准答案，员工提问时自动匹配推荐
- **FAQ管理（HR端）**：
  - 新增FAQ：录入问题（标题）+ 答案（富文本）+ 分类 + 关联制度文档
  - 编辑FAQ：修改问题和答案
  - 删除FAQ：逻辑删除（归档），不物理删除
  - FAQ分类管理：支持树形分类，同制度文档分类体系
  - 支持批量导入/导出FAQ（Excel格式）
- **FAQ匹配（员工端）**：
  - 员工输入问题时，实时匹配已有FAQ
  - 匹配算法：基于关键词匹配 + 语义相似度
  - 匹配结果展示：问题标题列表，点击查看完整答案
  - 匹配阈值：相似度 > 70% 自动推荐

#### FR-3.3 简单规则问答
- **描述**：基于正则表达式和关键词模板实现精准匹配问答
- **典型场景**：
  | 用户输入 | 匹配模板 | 返回结果 |
  |----------|----------|----------|
  | "年假几天" | `年假.*天` → 年休假规则模板 | 根据工龄返回梯度天数 |
  | "病假工资怎么算" | `病假.*工资` → 病假薪酬模板 | 返回病假期薪酬计算规则条文 |
  | "加班费标准" | `加班.*标准|加班.*费` → 加班薪酬模板 | 返回加班费计算标准 |
- **处理流程**：
  1. 对输入进行预处理（去停用词、标准化）
  2. 匹配预定义关键词模板库
  3. 命中的模板返回预设条文摘要
  4. 未命中则降级至FAQ匹配，再降级至全文搜索
- **模板维护策略**：
  - **一期**：以FAQ为主力匹配方式，简单规则模板作为补充（不独立实现规则模板管理界面）
  - **二期**：利用LLM自动从已上传制度文档中提取结构化Q&A对，自动生成关键词模板，HR审核后入库

### 4.4 高级智能问答模块（RAG）

#### FR-4.1 知识库向量化
- **描述**：将所有制度文档进行向量化处理，构建私有向量知识库
- **LLM方案**：采用**云端API调用**模式（如通义千问/文心一言/GLM API），低成本快速接入；通过统一 `LLMProvider` 接口封装，支持未来切换模型
- **处理流程**：
  1. 文档解析：提取纯文本，按段落/章节分块（Chunk Size ≈ 500 tokens）
  2. 文本向量化：调用Embedding模型将文本块转为向量
  3. 向量存储：存入向量数据库，建立索引
- **触发时机**：
  - 新文档上传后自动触发
  - 文档更新后对变更部分重新向量化
  - 支持手动触发全量重建索引

#### FR-4.2 精准回答生成
- **描述**：用户提问时，RAG流程检索相关制度片段并生成回答
- **处理流程**：

```
用户提问
    │
    ▼
① 问题向量化（Embedding）
    │
    ▼
② 向量检索 → Top-K相关制度片段
    │
    ▼
③ 构建Prompt：系统指令 + 检索片段 + 用户问题 + 对话历史
    │
    ▼
④ LLM生成回答（必须引用出处）
    │
    ▼
⑤ 返回：回答正文 + 参考条文链接
```

- **回答要求**：
  - **必须有出处**：每条回答附"参考：《XX制度》第X条"
  - **无法回答时**：明确提示"未找到相关制度，建议联系HR部门"，**严禁编造**
  - **置信度标识**：高（>85%）/ 中（70-85%）/ 低（<70%）
  - 低置信度回答需提示"以下回答仅供参考，建议与HR确认"

#### FR-4.3 多轮对话
- **描述**：支持上下文连贯的多轮追问
- **对话管理**：
  - 单个会话（Session）保留最近20轮对话上下文
  - 会话超时：30分钟无交互自动结束
  - 用户可主动开启新会话（清除上下文）
- **示例场景**：
  ```
  用户：年假怎么算？
  系统：根据公司休假制度，您的工作年限对应的年假天数为...
  用户：那我今年还剩几天？
  系统：（结合前文"年假"上下文+工龄信息）您今年年假共X天，已使用Y天，剩余Z天...
  用户：如果离职了年假能折现吗？
  系统：（维护"年假"上下文）根据制度第X条，离职时未休年假按日工资...
  ```

#### FR-4.4 跨文档综合推理
- **描述**：综合多份制度文档回答复合性问题
- **示例**：
  | 用户问题 | 涉及文档 | 综合回答 |
  |----------|----------|----------|
  | "加班到晚上10点，第二天可以晚到吗？还有加班费怎么算？" | 考勤制度 + 薪酬制度 | 综合加班调休规则和加班费标准 |
  | "婚假期间遇到法定节假日怎么算？婚假工资怎么发？" | 休假制度 + 薪酬制度 | 综合假期计算规则和假期薪酬规定 |

#### FR-4.5 条件判断问答
- **描述**：结合员工个人信息进行个性化条件计算
- **输入条件**（需获取员工信息）：
  - 入职时间 → 自动计算工龄
  - 职级 → 匹配对应福利标准
  - 地区（工作地）→ 匹配地方性政策差异
  - 家庭状况（婚否/育否）→ 匹配婚假/产假/育儿假资格
- **示例**：
  - 输入："我工作满3年了，年假多少天？"
  - 处理：识别工龄≥3年 → 查询年假梯度规则 → 返回"您的年假为X天"
  - 输入："我能休几天婚假？"
  - 处理：检查婚姻状态+工作地 → 返回婚假天数

### 4.5 制度解读与个性化报告模块

#### FR-5.1 制度智能解读
- **描述**：AI将冗长条文转化为通俗易懂的解读
- **解读形式**：
  - **图文解读**：将条款转化为分段要点 + 插图说明
  - **流程可视化**：将流程性制度（如请假流程、报销流程）转为流程图
  - **差异对比**：不同职级/地区的政策差异以对比表格展示
- **示例**：
  - 输入制度条文：年假梯度规则（满满一页文字）
  - 输出解读：工龄分段表格 + 计算公式 + 示例说明

#### FR-5.2 个性化权益报告
- **描述**：根据员工个人信息，自动生成专属福利权益清单
- **数据来源**（课程设计方案）：系统内置员工信息管理模块，管理员通过Excel批量导入或手动录入员工数据（职级、工龄、地区、家庭状况等）；预留HR系统API接口，未来可对接
- **报告内容**：
  - 年假天数及剩余天数
  - 体检资格（年度/入职体检等）
  - 补贴明细（餐补、交通、通讯等）
  - 弹性福利额度
  - 其他适用权益（商业保险、培训基金等）
- **报告格式**：标题为"您的20XX年可享受福利清单"，支持PDF导出和打印
- **触发方式**：员工主动生成 / 年度自动推送

### 4.7 反馈纠错模块

#### FR-7.1 答案质量反馈
- **描述**：每条回答下方提供反馈入口
- **反馈按钮**：
  - 👍 **有帮助**：一键点击，记录正向反馈
  - 👎 **无帮助**：点击后弹出可选原因选择（答案不准确 / 没有回答我的问题 / 来源链接失效 / 其他）+ 文字补充
- **数据用途**：
  - 统计回答质量指标
  - 低分回答自动标记，提醒HR优化FAQ

#### FR-7.2 制度内容纠错申请
- **描述**：员工发现制度条文可能过时或错误时，提交纠错申请
- **流程**：
  1. 员工在文档浏览页点击"纠错"
  2. 选中问题段落，填写纠错说明
  3. 提交申请，状态为"待审核"
  4. HR专员收到纠错通知，审核处理
  5. HR可选择：采纳并更新文档 / 驳回并说明原因
  6. 处理结果通知提交人

- **纠错状态流转**：
  ```
  待审核 ──→ 已采纳（文档已更新）
       └──→ 已驳回（附驳回原因）
  ```

### 8.3 安全性要求

| 要求 | 说明 |
|------|------|
| 传输加密 | 全站HTTPS，TLS 1.2+ |
| 密码存储 | bcrypt加盐哈希，不可逆 |
| 敏感数据脱敏 | 前端展示时：手机号显示 `138****1234`（保留前三后四）；身份证号显示前6位+后4位，中间用星号替代；邮箱显示 `u***@company.com` |

| 会话安全 | Token有效期2小时，支持刷新令牌续期；闲置30分钟自动登出 |
| 输入防护 | SQL注入、XSS、CSRF防护 |
| 数据备份 | 每日全量备份，保留30天 |

|------|------|

## 11. 实施分阶段计划

### 11.1 一期：基础能力建设（预计8周）

**目标**：搭建系统骨架，实现基础检索问答闭环

| 周次 | 任务 | 交付物 |
|:----:|------|--------|
| W1-W2 | 项目初始化、数据库设计、用户认证模块 | 用户可注册登录 |
| W3-W4 | 文档上传管理、分类标签体系、版本控制 | HR可管理文档 |
| W5-W6 | 全文搜索引擎、FAQ管理、简单规则问答 | 员工可搜索和问答 |
| W7-W8 | 个人问答中心（历史、收藏）、联调测试 | 一期完整功能 |

**🔗 一期→二期架构过渡预留设计**：

一期开发时同步做以下前置设计，确保二期RAG可平滑接入，避免重构：

| 预留项 | 一期实现方式 | 二期接入方式 |
|--------|-------------|-------------|
| **文档Chunk切分接口** | 文档上传解析后，同步调用预留的 `DocumentChunkService.split(document)` 方法；该方法在一期仅做空实现（返回原文不分块），但接口契约已定义 | 二期实现真正的Split逻辑（按500 tokens分块），已有文档批量重跑 |
| **Embedding抽象层** | 定义 `EmbeddingProvider` 接口（方法：`embed(text) → vector`），一期使用 `NoOpEmbeddingProvider` 返回空向量 | 二期切换为真实的 `OpenAIEmbeddingProvider` 或本地模型实现 |
| **向量存储抽象层** | 定义 `VectorStore` 接口（方法：`store/query/delete`），一期使用 `NoOpVectorStore` | 二期切换为真实的向量数据库实现（Milvus/ChromaDB/Pinecone等） |
| **问答策略链** | 一期实现 `FAQMatcher → RuleMatcher → SearchFallback` 策略链；策略链末尾预留 `RAGHandler` 槽位（一期跳过） | 二期在策略链末尾插入 `RAGHandler`，无需改动已有策略 |
| **文档状态字段** | `documents` 表预留 `chunk_count`、`embedding_status` 字段（一期默认为0/null） | 二期向量化后回填这些字段，标记处理状态 |

```
一期问答策略链：
  FAQ匹配 → 规则匹配 → 全文搜索 → [RAG槽位-二期接入]
                                    ↑
                              预留但一期跳过

二期问答策略链：
  FAQ匹配 → 规则匹配 → 全文搜索 → RAG智能问答
```
